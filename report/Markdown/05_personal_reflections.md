# 6 个人反思与收获

## 6.1 苏易文 - App/Common 层

### 6.1.1 负责内容

App/Common 层，包括公共类型、游戏状态、通知工具、应用组合根、生命周期管理和三类绑定。

### 6.1.2 个人反思与收获

本次项目中我主要负责 App/Common 层。开发过程中我最深的体会是：公共层不能什么都放，Common 层越克制，后续协作越轻松。

一开始我们曾经倾向于在 Common 层定义比较完整的命令和快照接口，例如把输入动作、Tick、状态快照和绑定接口都统一抽象出来。这个方案看起来“完整”，但实际联调时会发现它容易让 Common 层承担过多职责：View 的输入细节、ViewModel 的命令粒度、状态同步方式都会被提前固定下来。后来参考课程示例后，我把公共契约收敛为三类更稳定的内容：`types.h` 中的基础值类型、`game_state.h` 中的只读显示状态、`notification.h` 中的轻量事件通知。输入命令则由 ViewModel 通过 `std::function` 暴露，App 层负责注入给 View。这样 Common 层只保留真正需要跨层共享的内容，边界比之前清楚很多。

在 `GameState` 的设计上，我也更理解了“给 View 的状态”和“ViewModel 内部状态”之间的区别。View 需要知道玩家位置、血量、动作状态、镜头位置、敌人列表和 HUD 数据，但不应该知道敌人追踪速度、Boss 触发阈值、攻击命中范围、跳跃计时等内部规则。把这些模拟细节留在 ViewModel 内部，可以避免 View 因规则调整频繁改动；而把绘制必需的数据整理进 `GameState`，又能让 View 不需要到处拼接内部对象。这个过程让我认识到，公共数据结构不是越详细越好，而是要刚好表达跨层协作所需的信息。

App 层的 `GameApplication` 让我真正体会到组合根的价值。程序入口 `main.cpp` 只创建 `GameApplication` 并调用 `run()`，具体的 `QApplication`、`MainWindow` 和 `GameViewModel` 都被藏在 PImpl 里。这样公开头文件不会泄漏 Qt 和具体模块类型，也避免了其他层绕过 App 直接拿到内部对象。所有 properties / commands / notification 绑定都集中在 `GameApplication::Impl` 构造函数中完成，调试绑定问题时只需要看一个地方，项目结构也更容易向队友解释。

这次协作中另一个收获是，接口稳定性对多人开发非常重要。View 层和 ViewModel 层同学都依赖 Common/App 提供的边界，如果我频繁修改命名或数据结构，就会给他们带来额外适配成本。因此后期我更注意在改接口前先确认它是否真的属于公共契约，是否可以用更小的类型表达，是否会迫使其他层知道不该知道的细节。这个思考过程比单纯写代码更有工程实践意义。

总体来说，App/Common 层让我对 MVVM 有了更进一步的理解。一个好的分层不是把代码机械拆开，而是让每一层只看到自己应该看到的东西。Common 提供稳定语言，App 完成对象组装，View 和 ViewModel 才能相对独立地演进。后续继续完善项目时，我会更加重视公共接口的稳定性和跨层边界。

## 6.2 陈棋隆 - View 层

### 6.2.1 负责内容

View 层，包括 `MainWindow`、`GameWidget`、键盘输入、定时器、绘制、HUD 和覆盖层。

### 6.2.2 个人反思与收获

**对 MVVM 模式的理解从"知道"变为"用过"。** 课程上讲的 MVVM 概念——数据绑定、命令模式、属性通知——在本次项目中全部落地为具体代码。特别是从第一版 Qt 信号方案切换到第二版 `std::function` 注入方案的过程，让我真正理解了"用什么机制做绑定"比"绑了什么"更影响架构质量。Qt 信号虽然方便，但它把 MVVM 绑定路径和特定 GUI 框架耦合在一起；`std::function` 回调则是纯 C++ 机制，View 和 ViewModel 都只依赖标准库。

**View 层的约束比想象中更重要。** 开发初期，容易在 View 的绘制代码中顺手加入一些判断逻辑（如"如果玩家血量低于 20% 就闪烁红色边框"）。后来意识到这种判断属于 ViewModel 的职责——血量阈值的含义应该由规则层定义，通过 `GameState` 的字段传达给 View。此后严格遵守"View 只读取状态，不判断状态含义"的原则，所有颜色切换、闪烁频率、显示/隐藏条件都由快照中的字段决定。

**Draw 比想象的更有挑战。** 在没有图片素材的条件下用 `QPainter` 手绘全部游戏画面，需要综合考虑角色辨识度、颜色方案一致性、不同状态下的姿态差异、缩放适配等。这是一次对图形 API 基本功的集中锻炼。

## 6.3 邱安琪 - ViewModel 层

### 6.3.1 负责内容

ViewModel 层，包括 `GameViewModel`、`GameSimulation`、移动、攻击、敌人 AI、Boss 生成和胜负判定。

### 6.3.2 个人反思与收获

在本次项目中，我负责 ViewModel 层的核心规则引擎和状态同步逻辑。MVVM 中 ViewModel 与 View 解耦的思想让我能够专注于游戏推进逻辑，也使工程更容易扩展和维护。在与袁老师交流后，我纠正了早期设计中的一些误区，例如没有提前考虑扩展性，导致 ViewModel 和 View 几乎一对一绑定。工程一旦变复杂，这种做法就只能依靠不断增加绑定接口维持，后续修改会比较困难。

与此同时，Tick 驱动的推进方式让我进一步理解了游戏逻辑与渲染更新的关系：每一帧由 ViewModel 收集输入并调用 Simulation 进行一步推进，再把结果整理为统一的 GameState 快照，从而保证界面显示与内部规则始终保持一致。

由于之前没有相关经验，我先学习了状态机、碰撞判定和攻击盒等游戏逻辑，再一步步应用到项目中。看到代码最终变成可以实际操作的游戏，让我很有成就感。
