# 7 总结与展望

## 7.1 架构收获

本实验最重要的收获是通过 MVVM 把"输入、规则、显示"拆开。最初找袁老师验收时，我们的代码耦合度非常高：键盘事件里直接改坐标，绘制函数里顺手判断攻击，敌人 AI 和窗口状态混在一起。经袁老师指出，这样的代码很难并行开发，后续迭代和维护都会变得非常困难。

在袁老师的指导下，我们进行了四次改进，最终通过了袁老师的验收。当前实现将跨层共享内容收敛为 `GameState`、基础值类型和通知工具，并把命令设计为从 ViewModel 暴露出来的 `std::function` 回调。ViewModel 负责规则，View 只根据只读状态完成绘制。如此一来，每个人可以围绕自己的层次推进，不必频繁修改其他人的内部实现。

另一个收获是组合根的重要性。`GameApplication` 集中创建 Qt 应用、主窗口和 ViewModel，并在一个地方完成 properties / commands / notification 绑定。这样 main 函数、View 和 ViewModel 都不需要知道完整对象组装细节，依赖方向更加清楚。


## 7.2 项目整体进展的经验体会

回头看这段时间的开发，项目是从最开始只能打开窗口、显示简单画面，一点点做到现在可以移动、攻击、打 Boss，并且有暂停、失败、胜利和重新开始等完整流程的。虽然目前离一款内容丰富的游戏还有距离，但至少已经做出了一个可以实际游玩的原型。对我们来说，这个过程不只是把几个功能拼在一起，更重要的是把输入、游戏逻辑、状态更新和界面绘制真正连接了起来。

开发过程中比较明显的一个体会是，功能不能一下子铺得太多。刚开始我们先保证窗口、Tick 更新、角色移动和画面绘制能够正常工作，之后才陆续加入攻击、精力、敌人 AI 和 Boss 等内容。这样做虽然看起来慢一些，但每次出现问题时比较容易确定是哪一次修改造成的。如果一开始就同时做复杂战斗、动画和关卡，很可能一个错误会影响很多地方，反而更难调试。

项目的架构也不是一开始就完全确定的。虽然目录上采用了 MVVM 分层，但实际开发后还是出现过 Common 层内容太多、接口命名反复变化、View 和 ViewModel 绑定过于直接等问题。后来大家根据各自模块的实际需求重新整理接口，把 View 需要的数据统一放进 `GameState`，把具体规则留在 ViewModel，再由 App 层集中完成命令和通知的绑定。经过这些调整以后，各层负责什么、不能做什么都清楚了很多。我们也体会到，重构不一定代表前面的代码白写了，有时只有真正用起来，才能发现原来的设计哪里不合适。

多人合作方面，前期公共接口变化比较频繁，其他成员经常需要跟着修改，这部分确实产生了一些额外工作。后期我们开始更注意提前沟通字段含义和命令名称，并通过代码注释、README 和 Git 提交说明修改内容，联调效率比之前高了不少。如果以后再做类似项目，我们会尽量更早列出公共接口和阶段目标，同时增加相互检查代码的过程，而不是等到最后合并时才集中发现问题。

目前项目还有不少可以继续改进的地方，例如自动化测试比较少，输入、界面和跨层绑定大多还是依靠人工运行检查；角色动作、攻击反馈、音效和关卡内容也比较基础。不过，通过这次实验，我们已经实际经历了从分工开发到合并、调试和完善的完整过程，也更加理解了一个项目能够稳定运行，除了实现功能之外，还需要清晰的结构、及时的沟通和持续的检查。这些经验对我们之后继续完成项目会有很大帮助。




## 7.3 智能体的使用

本项目的需求分析、任务分工、架构选择、核心代码实现和最终联调均由三位成员共同完成。开发过程中我们使用了 Claude Code 智能体，但它的定位始终是辅助工具，而不是项目的设计者或完成人。每次使用前，都是由成员先明确要解决的问题和修改范围；智能体给出结果后，再由对应模块的负责人阅读、修改、编译和运行验证，确认没有问题后才会保留到项目中。

智能体主要用在以下几类工作中：

- **辅助处理重复代码。** 在接口已经由成员确定后，智能体可以协助补充部分 setter、getter、命令绑定和绘制代码骨架。它减少了重复输入的时间，但角色如何绘制、HUD 如何布局、命令应该怎样划分，仍然由负责对应模块的成员决定。
- **辅助搜索和重构。** Common 层类型或字段发生变化时，我们利用智能体查找旧名称的引用，并给出需要同步修改的位置。成员会逐项检查这些修改，避免误改其他变量或破坏原有逻辑。公共接口最终保留哪些内容，也是三位成员讨论后确定的。
- **辅助排查问题。** 遇到编译错误或跨层绑定问题时，智能体可以帮助整理可能原因，例如头文件依赖、函数签名不一致或命令遗漏等。不过，真正的问题需要成员结合代码、编译信息和运行结果定位，不能只根据智能体的回答判断。
- **辅助整理报告。** 智能体参与了部分章节初稿、代码说明和文字结构的整理。报告中的项目过程、成员分工、技术难点和个人体会均来自我们的实际开发经历，生成的文字也经过了成员修改，避免出现与代码不符或表达过于模板化的内容。

从实际效果看，智能体对批量修改、代码搜索和初稿整理比较有帮助，节省了一些机械性工作的时间，也能在遇到问题时提供新的排查思路。这样三位成员可以把更多精力放在 MVVM 分层、公共接口、游戏规则、画面效果和模块联调等需要人工判断的工作上。项目最终采用的架构、各层职责和游戏功能，都是成员在多次讨论、修改和验收后确定的，并不是直接使用智能体一次生成的结果。

在实际使用中，我们也发现了智能体存在以下问题：

- **命名不一定准确。** 智能体有时会根据局部代码直接取名，而没有充分考虑类型在整个项目中的含义。例如，它曾经倾向于把 Common 层中的 `GameState` 命名为 `GameViewState`。这个状态虽然会被 View 读取，但它本质上是 View 和 ViewModel 共同使用的跨层状态；加入 `View` 后容易让人误以为该类型只属于界面层，反而造成职责上的歧义。
- **不能很好地理解 MVVM 边界。** 智能体能够按照要求生成能运行的代码，但不一定真正理解各层为什么要这样划分。生成结果中曾出现把输入细节放入 Common、让 View 直接处理游戏规则，或者让 ViewModel 依赖界面类型等情况，导致比较严重的层间耦合。这些问题通常不会马上表现为编译错误，而是在后续修改和联调时才暴露出来。
- **容易出现便宜模式。** 智能体为了尽快完成当前要求，有时会直接暴露内部对象、增加临时接口或绕过原有分层。这样的代码可能暂时解决问题，但会破坏依赖方向，并把维护成本留到后面。因此，不能只根据“现在能不能运行”判断生成结果是否合适。
- **复杂逻辑容易越写越绕。** 当提示中同时包含较多条件时，智能体可能不断增加分支、状态变量和补丁式判断，最后形成层层嵌套、重复判断较多的逻辑。虽然功能表面上能够实现，但代码可读性和可维护性较差，需要成员重新梳理数据流、拆分函数，有时甚至需要重新设计后再实现。
- **对使用者的意见过于顺从。** 当我们提出一个方案时，智能体往往会先表示认可，再沿着这个方向继续生成代码，即使方案本身可能存在架构或逻辑问题。它较少主动质疑需求中的错误前提，也不一定会比较不同方案的代价。因此，成员不能把智能体的认同当作方案正确的证明，而应结合课程要求、代码结构、运行结果和团队讨论独立判断。

正因为存在这些问题，我们始终把智能体的输出当作参考或初稿，而不是可以直接使用的最终答案。架构边界、类型命名、功能逻辑和最终代码都由三位成员共同检查，并通过编译、运行、试玩和老师验收不断修改。智能体提高了部分工作的效率，但项目质量最终仍然取决于成员自己的理解、判断和协作。


## 7.6 后续改进方向

当前项目已经具备核心玩法骨架，但仍有一些可以继续完善的方向：

- 将敌人、Boss 触发和战斗数值从硬编码常量抽离成配置表，方便后续调整难度。
- 增加更完整的连招链、击退、击倒、无敌帧和受击硬直。
- 补充遭遇战锁屏、清屏解锁和 GO 提示字段，使关卡推进更接近最终设计。
- 引入图片、精灵动画和音效资源，替代当前的基础 `QPainter` 几何绘制。
- 补充单元测试，覆盖移动边界、攻击命中、Boss 生成、胜负判定等规则。
- 增加攻击命中的画面反馈——屏幕震动、受击粒子、伤害数字飘字。


## 7.7 总结

本阶段完成了一个可运行且架构正确的横版动作游戏原型。项目从工程组织、公共状态、命令回调、界面绘制到规则模拟都已经形成闭环。通过 MVVM 分层和 `std::function` 命令注入，View 和 ViewModel 保持了清晰的单向依赖，为后续功能扩展和界面升级奠定了扎实的架构基础。
